### Post 20 — Server Components Changed Everything

React Server Components are the biggest paradigm shift in frontend development since React itself.

**Before:** Your server renders HTML → sends it with a massive JavaScript bundle → browser re-hydrates everything → page becomes interactive. Users wait.

**After:** Server renders components that don't need interactivity on the server. Only interactive components send JavaScript. Dramatically less JS shipped to the browser.

Real impact seen across production projects:
→ 60% reduction in JavaScript bundle size
→ 40% improvement in Time to Interactive
→ Better SEO from true server rendering
→ Simplified data fetching (no more useEffect chains)

The mental model shift: Stop thinking "frontend vs. backend." Start thinking "what needs to be interactive vs. what doesn't."

Most of your UI doesn't need JavaScript. Server Components finally let you act on that truth.

Are you using Server Components yet?

### Post 66 — Testing Strategy > Test Coverage

**End-to-End Tests (10%):** Critical user journeys. Run before deployment.

→ Framework code (React, Next.js already tested)
→ Simple getters/setters
→ Implementation details (test behavior, not internals)

The metric that matters isn't coverage percentage. It's: "When a bug reaches production, was there a test we should have written?"

### Post 70 — Observability > Monitoring

Monitoring tells you WHAT is broken.
Observability tells you WHY.

**Monitoring (the basics):**
→ CPU usage is high
→ Error rate is up
→ Response times are slow

**Observability (the insight):**
→ CPU is high BECAUSE a specific query is running a full table scan
→ Error rate is up BECAUSE a third-party API changed its response format
→ Response times are slow BECAUSE connection pool exhaustion is causing request queuing

The three pillars:
1. **Logs** — What happened (structured, not free-text)
2. **Metrics** — How much is happening (time-series data)
3. **Traces** — The path a request takes through your system

Together: the ability to ask questions you didn't think to ask before the incident.

That's observability. The difference between firefighting and engineering.

### Post 72 — Error Handling Is Not An Afterthought

→ Swallow errors silently
→ Show stack traces to users
→ Use generic "Something went wrong" for everything
→ Retry infinitely without backoff
→ Log errors without context

→ Categorize errors (client vs. server vs. external dependency)
→ Provide actionable error messages
→ Implement exponential backoff for retries
→ Add circuit breakers for external dependencies
→ Include request IDs for traceability

The user's experience of your software is defined by how it handles failure, not how it handles success.

Because success is easy. Failure is where trust is built or broken.

### Post 98 — A Day in Engineering Life

What a typical day looks like for a high-performing engineering team:

**9:00** — Morning sync. 15 minutes. Blockers, priorities, progress. No status reports.

**9:15** — Deep work block. 3 uninterrupted hours. Slack on DND. No meetings.

**12:15** — Lunch + learning (optional). Tech talks, paper discussions, demo sessions.

**13:00** — Collaboration window. Code reviews, pair programming, architecture discussions, client calls.

**15:00** — Second deep work block. Another focused window for complex work.

**17:00** — Async wrap-up. Update tickets, document decisions, prepare tomorrow.

The principles:
→ Protect deep work (context switching kills productivity)
→ Batch meetings (don't scatter across the day)
→ Async by default, sync when needed
→ No after-hours expectations

The best engineering happens when people have time and space to think deeply.
